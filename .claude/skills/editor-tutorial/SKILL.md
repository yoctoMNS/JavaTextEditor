# editor-tutorial（㉒）— vimtutor 形式の対話型チュートリアル

## このスキルが解決すること

「このエディタを使いこなせるようになりたい初心者」のために、`:tutor`（`:tutorial` でも可）コマンドで開く対話型チュートリアルを提供する。Vim 本家の `vimtutor` と同じ設計思想——**読み物ではなく、実際にそのテキストを編集しながら体で覚える**——を踏襲する。

対象は Vim 標準のモーダル編集だけでなく、このエディタ独自の機能（Emacs式INSERT移動、Spaceリーダーキー、Java編集支援、プロジェクト全体機能、FILERモード等）も含む。「Vim のチュートリアル」ではなく「**このエディタの**チュートリアル」である点が vimtutor との違い。

---

## なぜ自動採点をしないのか（最重要の設計判断）

vimtutor は「この行の xxxxx を x で消してください」のような指示を出すだけで、実際にユーザーが正しく操作したかを判定する仕組みを持たない。本プロジェクトもこれを踏襲した。

**理由**:
1. 自動採点を実装するには「チュートリアル専用の Mode」「各レッスンの期待状態の定義」「キー入力ごとの差分判定」が必要になり、実装・保守コストに対して学習効果の向上が見合わない。
2. CLAUDE.md の「無駄を削ぎ落としたシンプルな環境」という方針に反する。
3. vimtutor 自体が何十年も「自動採点なし」で教育効果を上げてきた実績があり、後発で同じ路線を踏襲するのが合理的。

**結果として**: チュートリアルは「`:enew` に近い、ただし最初から内容が入っている通常のバッファ」として実装される。ユーザーがチュートリアル中に行う `x`・`dd`・`yy`・`p`・`u` 等の操作は、すべて本物の編集操作としてこのバッファに対して働く。これは利点でもある——「練習用の特別なサンドボックス」ではなく「本物のエディタ」で練習するため、操作の手触りが実際の編集と完全に一致する。

## 本文の持ち方: Java 21 テキストブロックへの埋め込み

`scripts/build.sh` は `find src -name "*.java" | xargs javac ... -d build` のみを行い、`.txt` 等の非 `.java` リソースを `build/` にコピーする処理を持たない（CLAUDE.md 既定のビルド方針: ビルドツール不使用・`javac` 直接呼び出し）。したがって、チュートリアル本文を外部テキストファイルとして `lib/` 等に同梱する方式は採れない。

→ `dev.javatexteditor.tutorial.Tutorial` クラスの `public static final String CONTENT` に、Java 21 のテキストブロック（`"""..."""`）としてそのまま埋め込む。

```java
public final class Tutorial {
    private Tutorial() {}
    public static final String CONTENT = """
==============================================================================
=          JavaTextEditor チュートリアルへようこそ                          =
==============================================================================
...
""";
}
```

### テキストブロック使用時の注意（実装時に踏んだ罠）

本文中で `\f`（ファイル名検索キー）や `\g`（内容grep検索キー）をバックスラッシュ付きの文字列として表示したい場合、テキストブロックでも通常の文字列リテラルと同じエスケープ規則が適用されるため、`\f` はフォームフィード文字、`\g` は**不正なエスケープシーケンスとしてコンパイルエラー**になる。リテラルな `\f` という2文字を出力するには `\\f` と書く必要がある。

```java
// NG: \f はフォームフィード文字になってしまう（表示が壊れる）
// NG: \g は invalid escape character でコンパイルエラー
  \f                       ファイル名検索

// OK: \\f と書けば、出力されるテキストは "\f"（バックスラッシュ+f）になる
  \\f                       ファイル名検索
```

## `:tutor` コマンドの実装方針: 既存の疑似バッファパターンを再利用する

`ModalEditor` には `:enew`（`newBuffer()`）・`:grep`（`executeGrep()`）・`:rename`（`executeRename()`）が共通して使う「現在のバッファを履歴に積んでから、新しい内容の `UndoablePieceTable` に差し替える」というパターンが既にある。`:tutor` もこれを踏襲する。

```java
private void openTutorial() {
    pushBuffer();                                   // 現在のバッファを履歴に積む
    buffer = new UndoablePieceTable(Tutorial.CONTENT);
    currentFilePath = null;                          // 保存先を持たない疑似バッファ
    cursorRow = 0;
    cursorCol = 0;
    grepResults = null;
    fileNameResults = null;
    searchMatches = List.of();
    currentMatchIdx = -1;
    statusMessage = "チュートリアルを開きました — :q で終了、Ctrl+U で元のバッファに戻れます";
}
```

`executeCommand(String cmd)` 内のディスパッチに以下を追加するだけで良い（新しい Mode は不要）:

```java
} else if (cmd.equals("tutor") || cmd.equals("Tutor") || cmd.equals("tutorial")) {
    openTutorial();
}
```

### `currentFilePath = null` の意味

`:w` で保存しようとすると `saveToFile()` が `path == null` を検出して `"E: no file name"` エラーを返す。これは `:enew` で新規バッファを開いた場合と全く同じ挙動であり、特別なハンドリングは不要——チュートリアルバッファを誤って意図しないパスに保存してしまう事故を、既存のエラーメッセージがそのまま防いでくれる。

### 元のバッファへの復帰

`pushBuffer()` で履歴に積んでいるため、既存の `Ctrl+U`（前のバッファへ）/ `Ctrl+P`（次のバッファへ）がそのまま使える。チュートリアル専用の「戻る」キーを新設する必要はない。

## 章立て（実装済みレッスン構成）

`Tutorial.CONTENT` は以下の16レッスン構成（vimtutor の章立てを参考にしつつ、後半をこのエディタ独自機能に充てている。全機能を網羅する目的で 2026-07 に13→15→16レッスンへ段階的に拡張した）:

| # | テーマ | 範囲 |
|---|---|---|
| 1 | モードという考え方 | NORMAL/INSERT/VISUAL/COMMAND の概念、Escape の役割 |
| 2 | カーソル移動の基本 | `h j k l` |
| 3 | 単語移動・行頭/行末・ファイル先頭/末尾 | `w b e 0 ^ $ gg G` |
| 4 | 削除とアンドゥ | `x dd u Ctrl+R` |
| 5 | 挿入モード | `i a o Escape` |
| 6 | ヤンクとペースト | `yy dd p P` |
| 7 | VISUAL モード | `v V y d`・単語/行頭行末での範囲拡張・`>`/`<` インデント |
| 8 | VISUAL BLOCK モード | `Ctrl+V`・`I`/`A`/`c`/`r`・矩形ヤンク/貼り付け |
| 9 | 文字列検索 | `/ n N * #` |
| 10 | コマンドモード | `:w :e :q :wq`・`:sp`/`:vs` 画面分割 |
| 11 | 置換コマンド | `:s`/`:%s`/`:'<,'>s`/`:N,Ms`・`g`/`i`フラグ・`\1`/`&`置換 |
| 12 | エディタ独自機能（1）移動・効率化キー | Emacs式INSERT移動・Spaceリーダー・`Ctrl+F/B/D/E/Y`スクロール・`H/M/L`・`%`・`Alt+J`/`Alt+K`・`sv`/`ss`/`sh`/`sl` |
| 13 | エディタ独自機能（2）Java編集支援 | `K`（統合ジャンプ）・`Shift+J`・`gr`・`Ctrl+Space`/`Alt+/`補完・`[g`/`[d`・`F2`・Getter/Setter生成・`Ctrl+Shift+O`・`:main`・auto-import |
| 14 | エディタ独自機能（3）プロジェクト全体機能 | `:grep`/`:grep!`・`:rename`・`\f`/`\g`（bang版含む）・`gR`・telescope・`:cd`/FILERモード・TAB補完 |
| 15 | エディタ独自機能（4）見た目・その他 | フォントセルサイズ調整・ステータスバーの時計/システムステータス/アニメーション |
| 16 | 困ったときは | `u`・Escape・`Ctrl+U`/`Ctrl+P`・`:q` のまとめ |

各レッスンは「説明」＋「練習」（実際に編集する具体的な指示とダミーテキスト）の2部構成。練習用のダミーテキストはレッスンの説明文のすぐ下に配置し、ユーザーがスクロールして探す手間を最小化している。詳細な機能説明そのものは `docs/manual/` に譲り、チュートリアル本文は「操作を体で覚えるための最小限の説明＋練習」に留めている。

## テスト方針

自動採点をしないため、テストの主眼は「①本文に必要なレッスンが揃っているか」「②`:tutor`コマンドが正しく疑似バッファパターンに乗っているか」の2点に絞られる（レッスンの教育的妥当性そのものはテストでは検証できない）。

```
test/dev/javatexteditor/tutorial/TutorialTest.java
  ├── Tutorial.CONTENT 単体テスト
  │     ├── 空でないこと
  │     ├── レッスン1〜13の見出しがすべて含まれること
  │     └── \\f / \\g のエスケープが正しくリテラル文字列になっていること
  │         （= フォームフィード文字が紛れ込んでいないこと）
  └── ModalEditor 統合テスト（sendCommand ヘルパーで ":tutor" を送信）
        ├── バッファにチュートリアル本文が読み込まれること
        ├── currentFilePath が null になること
        ├── カーソルが (0,0) にリセットされること
        ├── Ctrl+U で元のバッファに戻れること（pushBuffer の確認）
        ├── :tutorial エイリアスも同様に動くこと
        └── 開いた直後のバッファが本当に編集可能（x で1文字消せる）であること
```

9 テスト全 PASS。

## 次に学ぶべきこと・拡張時の注意

- レッスンを追加する場合は `Tutorial.CONTENT` にセクションを追記し、`TutorialTest.testContentHasAllLessons()` のレッスン番号チェックも更新すること。
- 将来、進捗保存（「前回どこまで進んだか」を `Preferences` に記録する等）を検討する場合は、まず本当に必要かを再検討すること——vimtutor が進捗保存なしで成立している以上、複雑化に見合う効果があるかを優先的に疑うこと。
- 本文中にこのエディタのキーバインドを追記・変更する場合、README.md の「キーバインド」表と内容がズレないよう、変更のたびに両方を確認すること。
