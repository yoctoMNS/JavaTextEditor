# 要件定義書（draft v1）

最終更新: 2026-06-24　※この文書はまだドラフトです。「未確認」「TODO」と書かれた箇所は次回以降の相談で確定させること。

## 1. プロジェクトビジョン

- 普段使いのキーバインドはVim（NeoVim）。過去にEmacsを1年程度使用した経験があり、Linuxのターミナルエミュレータでは現在もEmacsキーバインドを採用している。
- Emacsの「編集モードを変えずにカーソル移動できる」操作感を気に入っており、これをVimの編集モデルの中（特にInsertモード）でも使えるようにしたい。
- 既存のJava向けIDEは「重い」「機能が大げさ」と感じている。10年のJava開発経験があり、IDEの補助機能に依存せずコーディングを楽しみたい。
- 無駄な機能を削ぎ落とした、シンプルなJavaプログラミング環境を志向する。
- エディタ本体もJavaのみで実装する（学習目的と実用目的の両立）。
- JDK内部構造（JNI・HotSpot）への学習・研究目的も併せ持つ。

## 2. 対象ユーザー・利用シーン

- 主な利用者: 開発者自身（経験豊富なJavaプログラマ）。
- 利用シーン: 日常のJavaコーディング、およびOpenJDK内部構造の学習・研究。

## 3. キーバインド方針

- 基本はVim（NeoVim）のモーダル編集（ノーマル/インサート/ビジュアル/コマンドラインモード）を土台にする。
- **追加要件**: Insertモード中でも、Emacs式のカーソル移動（例: 前後移動・行頭行末移動などのキー操作）を使えるようにする。
- → `keymap-conflict-resolution` Skillの中核要件として扱う。

## 4. 機能要件

### 4.1 エディタ基盤

| 機能 | 方針 |
|---|---|
| バッファ構造 | ピーステーブル方式（Java SE標準APIのみ）。数百行〜数十万行規模に対応 |
| モーダル編集 | Vim式のノーマル/インサート/ビジュアル/コマンドラインモード |
| Insertモード中のカーソル移動 | Emacs式キーバインドを追加で利用可能にする |
| 拡張性 | Lispではなく、JDK標準の`javax.tools.JavaCompiler`によるJavaソースの動的コンパイル・実行 |

### 4.2 Java開発支援機能（必須）

| 機能 | 動作方式 | 想定する技術基盤 | 備考 |
|---|---|---|---|
| 自動Import挿入・削除 | **リアルタイム**（入力中に自動補完・挿入） | Compiler Tree API（`com.sun.source.tree`等）によるAST解析 ＋ 索引キャッシュ（詳細は4.2.1） | 索引づくりと、入力中の検索を分離することで巨大ファイルでも軽快に動作する |
| javacによる自動コンパイル・エラー表示 | **保存時に自動実行**＋常に手動トリガーも可能 | `javac -sourcepath <作業ディレクトリ> <編集中のファイル>`形式で実行 | 単体ファイルだけを渡すと依存先クラスが解決できず誤検知するため、`-sourcepath`で作業ディレクトリを指定し、依存先は自動解決・無関係なファイルは対象外とする |
| JDKクラス/メソッド/フィールド参照（コードナビゲーション） | **手動トリガーのみ**（情報表示・ジャンプの両方とも、明示的なキー操作でのみ発生） | リフレクション or Compiler Tree API。ジャンプ先はJDK同梱の`lib/src.zip`内のソース（ネットワーク不要） | Vimの`K`（ドキュメント表示）やタグジャンプに相当する操作感を想定 |
| Oracle JavaSE APIドキュメント参照 | ローカルJDK同梱のJavadoc(HTML)をエディタ内に表示 | `JEditorPane`+`HTMLEditorKit`（要: 6章のリスク参照） | オフライン動作。事前にJavadoc生成が必要 |
| OpenJDK JNI/HotSpotレベルトレース | nativeメソッド検出 → ローカルOpenJDKソースをパターン検索 → 候補提示・ジャンプ | JNI命名規則（`Java_パッケージ_クラス_メソッド`）に基づく文字列検索 | 完全な網羅はできない（6章のリスク参照） |

#### 4.2.1 自動Importの索引設計（⑧`java-source-analysis`の中核方針）

リアルタイム性と巨大ファイル対応を両立させるため、「索引づくり」（重い・低頻度）と「索引を調べる」（軽い・リアルタイム）を分離する。

| 索引の種類 | 構築タイミング | 構築方法 |
|---|---|---|
| JDKクラス索引 | エディタ起動時に1回だけ | `jrt:/`という標準URIスキーム（`java.nio.file.FileSystems`、Java SE標準機能）でJDKのモジュール内クラス一覧を列挙する。JDK自体は実行中に変化しないため、再構築は基本的に不要 |
| 作業ディレクトリ索引 | 起動時に1回全体スキャン。以後はファイル保存時に該当ファイル分だけ差分更新 | 作業ディレクトリ配下の`.java`ファイルを走査し、各ファイルが定義するクラス名（パッケージ＋クラス名）を抜き出す |

**リアルタイム部分**は、上記2つの索引が完成している前提で、「今編集中の1ファイルだけ」をAST解析し、未解決の識別子を見つけて索引に問い合わせるだけの軽い処理にする。

**索引の共有範囲**: 索引は作業ディレクトリ（プロジェクト）単位で1つだけ保持し、ウィンドウ分割で複数のバッファを同時に開いていても、すべてのウィンドウが同じ索引を参照する（ウィンドウごとに別々の索引は持たない）。

**名前の衝突時の動作**: JDK側と作業ディレクトリ側で同じ単純名のクラスが見つかった場合（例: 自作の`List`クラスと`java.util.List`）、自動では選択せず、**小さな選択UIを表示してその場でユーザーに選んでもらう**。

### 4.3 スコープ外（確定）

**既存の決定から自動的に除外されるもの:**
- Maven/Gradleプロジェクトのインポート対応（ビルドツール不使用の方針から）
- 既存のVim/Emacsプラグイン（VimScript/Emacs Lisp）との互換性（設定・拡張をJavaで書く方針から）

**今回の協議で明確に「やらない」と決定したもの:**
- デバッガ統合（ブレークポイント・ステップ実行）
- IntelliSense的な一般コード補完（メソッド名等の入力補助）
- マルチプロジェクト/ワークスペース管理（ファイルツリー等のサイドバーUI）
- Git/バージョン管理の統合
- クラウド同期・リモート編集（SSH等）
- コードスニペット・テンプレート機能
- プラグインの共有・マーケットプレイス機能
- 自動アップデート機能
- マウス操作の充実（ドラッグ選択・右クリックメニュー等の高度な操作。クリックでの基本的なカーソル移動とは別に扱う）
- エディタUI自体の多言語対応

**設計思想（確認待ち）**: 上記の選び方から、「経験者向けの入力補助（補完・デバッガ）は不要だが、テキスト操作の強力な道具（検索・リファクタリング）は必要」という方針が読み取れる。次節の項目はこの方針のもとで検討中。

### 4.4 プロジェクト全体に関わる機能（確定）

「プロジェクト」は、エディタ起動時の**作業ディレクトリ（カレントディレクトリ）以下**を暗黙の対象範囲とする。ファイルツリーのようなサイドバーUIは出さないが、内部的にはこの範囲をスキャンする。

- 技術的な実現方法: `System.getProperty("user.dir")`（Java SE標準機能）で作業ディレクトリを取得し、その配下を対象にする。
- TODO: ディレクトリ走査時に`.git`やビルド出力ディレクトリ等を除外するかは別途決定する。

| 機能 | 必要な技術基盤 | 依存関係 |
|---|---|---|
| プロジェクト全体検索（grep的機能） | 単純な文字列/パターン照合 | ①のみに依存。独立して着手可能 |
| 複数ファイルをまたぐリファクタリング | シンボル解決（識別子がどの定義を指すか） | ⑧`java-source-analysis`に依存 |

Vimの`:grep`/`:vimgrep`コマンドとquickfixリスト（検索結果一覧からジャンプできる仕組み）に相当する操作感を想定する。

## 5. UI要件

- 描画方式: GUIアプリケーション（JFrame + Canvas、`Graphics2D`による直接描画）。ターミナル（TUI）ではない。
- カラーテーマ:
  - ライトモード: ベージュ背景 / 薄い黒文字（デフォルト）
  - ダークモード: 黒背景 / 薄いグレー寄りの白文字
  - **設定方法**: `TextEditorSettings.java`という通常のJavaソースファイル（`src/`配下、他のファイルと同様に`javac`でビルドする。③の動的コンパイル機構は不要）に、`Theme.LIGHT_MODE`または`Theme.DARK_MODE`という定数を指定することでプリセットを適用する。個別の色を上書きしたい場合はフィールドを直接指定することも可能。
  - 設定変更には再ビルド・再起動が必要（両プリセットの配色情報はビルド時に両方ともコンパイルされるため、起動後にキー操作で2プリセットを動的に切り替えることも追加コストなく可能。要確認）
- フォント: OS標準のモノスペースフォント（`Font.MONOSPACED`論理フォント）を使用。追加ファイルの同梱は行わない。見た目はOS依存となる。
- ウィンドウ分割: **持つ**。複数バッファを並べて表示できる。実現にはSwing標準の`JSplitPane`を使用する想定。
  - **設計原則（②への前提）**: モード（NORMAL/INSERT/VISUAL等）はエディタ全体で1つの状態として持つ。ウィンドウ（ペイン）ごとに個別管理するのはカーソル位置とスクロール位置のみ。
- 画面解像度（HiDPI/Retina）対応: 現時点では特別な対応をせず、JDK標準のスケーリング挙動に任せる。見た目の問題が生じた場合に再検討する。
- モード表示: NORMAL/INSERT/VISUAL等の現在のモードを示すステータス行を画面内に設ける。

## 6. 既知の技術的リスク

1. **JavadocのHTML表示制限**: Swing標準の`JEditorPane`はHTML 3.2程度までしか正しく描画できない。JDK 9以降の公式Javadocテーマ（検索機能付き）は現代的なCSS/JavaScriptを使うため、そのままでは正しく表示されない可能性が高い。対策は別途検討する。
2. **JNI/HotSpotパターン検索の限界**: 標準のJNI命名規則に従う実装は検索で発見できるが、HotSpot内部の一部の高速化されたメソッド（`register_natives`による動的登録、JITコンパイラのintrinsic最適化など）は規則に従わないため、パターン検索だけでは原理的に発見できない。これは機能の限界として明示する。
3. **リアルタイムauto-import × 巨大ファイルの両立**: 4.2.1節の設計（索引づくりと、入力中の検索の分離）により対応済み。ただし、作業ディレクトリ内のファイル数が極端に多い場合、起動時の初回スキャンに時間がかかる可能性があり、その場合はスキャンの進捗表示や非同期化が必要になる。

## 7. 技術的制約（既決定・CLAUDE.mdと同期）

- Java 21 (LTS) / Java SE標準APIのみ / 外部ライブラリ不使用
- ビルド: `javac`直接呼び出し、ビルドツール（Maven/Gradle）不使用
- テスト: 自作テストハーネス（`main`メソッド方式）、JUnit不使用
