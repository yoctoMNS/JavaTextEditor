# 11. C言語開発支援

このエディタは Java だけでなく **C言語**（`.c` / `.h`）の開発も支援します。Java 側が
`javax.tools.JavaCompiler`（JDK標準API）でコンパイル・診断を行うのに対し、C には相当する
標準APIが存在しないため、C 側は **外部の C コンパイラを子プロセスとして起動** して実現します
（`F11` の Java 実行がすでに `java` を別プロセスで起動しているのと同じ方式です）。

コンパイラは **`gcc` → `clang` → `cc`** の順に PATH から自動検出されます。どれも見つからない環境では
C の各機能は静かに無効化されます（診断は表示されず、エディタ本体は問題なく動作します）。

## 対象ファイルの判定

`.c` または `.h` を開いているとき（`currentFilePath` の拡張子で判定）に C 用の動作が有効になります。
`:enew` 等でファイルパスが未設定の疑似バッファは対象外です。`.java` を開いていれば Java 用の動作に
なります（[Java開発支援](04-java-tooling.md) を参照）。

## プロジェクト全体のコンパイル・実行（`F10` / `F11` / `F12`）

`.c` ファイルを開いた状態で以下のグローバルキー（NORMALモードのみ）を押すと、C としてコンパイル・
実行します。結果は Java と共通の `*compile*` / `*run*` 疑似バッファに表示されます。

| キー | 動作 |
|---|---|
| `F10` | 作業ディレクトリ配下の全 `.c` を **1つの実行ファイル** にコンパイル・リンクする |
| `F11` | コンパイル済みの実行ファイルを実行する |
| `F12` | `F10` を実行し、成功した場合のみ続けて `F11` を実行する |

- **コンパイル**: `gcc -Wall -o <bin>/a.out <全ての.c>` 相当のコマンドを発行します。実際に発行した
  コマンド文字列は `*compile*` バッファの先頭行に表示されます。`.git`/`build`/`target`/`.gradle`/
  `node_modules`/`.idea`/`.vscode`・出力先の `bin/` はスキャン対象から除外されます
- **出力先 `bin/` の場所**: Java版と同じ `ProjectBuilder.binDirFor()` の規則で決まります。`projectRoot`
  から祖先ディレクトリを遡り、`src` を直下に持つ最初のディレクトリ配下の `bin/`（＝ `src` と兄弟の位置）に
  実行ファイル `a.out` を出力します。どの祖先にも `src` が無ければ `projectRoot/bin` にフォールバック
  します（`:pr` でビルドルートを固定している場合はそちらが基準になります）
- **実行**: 生成された実行ファイル（`bin/a.out`）を別プロセスとして起動します。標準出力・標準エラーは
  それぞれリアルタイムに `*run*` バッファへ表示され、標準エラー由来の行は赤字になります。前回の実行
  プロセスが生きていれば強制終了してから起動し直します（多重実行防止）。`F11` は実行ファイルが未生成
  （＝ `F10` 未実行）の場合はエラーを表示して実行しません
- コンパイル・実行はいずれもバックグラウンドスレッドで行われ、UIをブロックしません

> **複数 main の制約**: 全 `.c` を1つの実行ファイルにリンクするため、それぞれ `main()` を持つ独立した
> 複数プログラムが同一 `projectRoot` 配下にあると、リンカが `multiple definition of main` エラーを
> 返します（このエラーはそのまま `*compile*` バッファに表示されます）。「1つのプログラム＝複数ファイル」
> という典型的な構成を主対象とした、意図的な仕様です。
>
> **標準入力**: 出力はリアルタイム表示しますが、`scanf` 等で標準入力を要求する対話的なプログラムは
> 正しく動作しません（Java版の `F11` と同じ既知の制約です）。

## コンパイルエラー・警告のインライン表示

`.c` バッファでは、保存時および INSERT→NORMAL 復帰時に `gcc -fsyntax-only -Wall` で解析され、
エラー・警告がガター（左端の `E`/`W`）と波下線で表示されます。これは Java の
[コンパイルエラー表示](04-java-tooling.md#コンパイルエラー表示) とまったく同じUI・同じキーで扱えます。

| キー | 動作 |
|---|---|
| `[g` | 次のエラー/警告行へジャンプ |
| `[d` | 前のエラー/警告行へジャンプ |
| `F2` | カーソル行のエラー・警告一覧をダイアログ表示 |

同一ディレクトリのローカルヘッダ（`#include "foo.h"`）を解決できるよう、編集中ファイルの親ディレクトリが
`-I` の探索パスに追加されます。

## `#include` の自動挿入（auto-import の C 版）

Java の auto-import に相当する機能です。`printf` / `malloc` / `strlen` / `sqrt` / `size_t` のような
標準ライブラリのシンボルを使っているのに対応するヘッダを `#include` していない場合、必要な標準ヘッダを
自動的に追加します。

| きっかけ | 動作 |
|---|---|
| INSERT→NORMAL 復帰・保存時 | `gcc` が報告した未定義シンボル（`implicit declaration of function` / `unknown type name` / `undeclared`）に対応するヘッダを自動挿入する |
| `Space i o` / `:oi` | ソース全体を走査し、使用している標準シンボルのうち未 `#include` のヘッダをまとめて追加する（`gcc` 不要） |

挿入位置は「既存の最後の `#include` の直後」（既存の include が無ければ先頭のコメント群の直後）で、
追加行はアルファベット順に整列します。対応しているヘッダの例:

- `<stdio.h>`: `printf` / `scanf` / `fopen` / `FILE` / `EOF` …
- `<stdlib.h>`: `malloc` / `free` / `atoi` / `qsort` / `EXIT_SUCCESS` …
- `<string.h>`: `strlen` / `strcpy` / `memcpy` …
- `<math.h>`: `sqrt` / `pow` / `sin` / `M_PI` …
- `<ctype.h>` / `<time.h>` / `<stdbool.h>` / `<stddef.h>` / `<stdint.h>` / `<assert.h>` / `<errno.h>` / `<unistd.h>`（POSIX）ほか

## もともと言語に依存しない機能

以下の機能は特定言語に依存せず、`.c` / `.h` ファイルでもそのまま使えます（C専用の対応は不要でした）。

| 機能 | 参照 |
|---|---|
| コード補完（`Ctrl+Space` / `Alt+/`） | [コード補完](05-completion.md)。`Alt+/` の単語索引は元々 `.c`/`.h`/`.cpp` を走査対象に含みます |
| プロジェクト全体検索（`:grep` / `\g` / `gr`） | [検索・ナビゲーション](03-search-and-navigation.md) |
| シンボルのリネーム（`:rename`） | [Java開発支援](04-java-tooling.md#マルチファイルリファクタリングrename)（言語非依存の語境界置換） |
| モーダル編集全般（NORMAL/INSERT/VISUAL/マクロ/置換など） | [モーダル編集](02-modal-editing.md) |

## Java にあって C にない機能

以下は Java（JDK）固有の概念のため、C には対応する機能がありません。

- Getter/Setter 自動生成（`Space g g/s/d`）
- JDK API参照・Javadocビューア（`K`）、OpenJDK native トレース
- `:main java` / `:main javac`（Javaランチャの起動点へのジャンプ）

C の識別子に対する `K`（定義ジャンプ）は、プロジェクト全体の正規表現検索（`gr` と同じ基盤）で
部分的に働きますが、型解決を伴う厳密なジャンプ（Java の binding 解決相当）は行いません。
